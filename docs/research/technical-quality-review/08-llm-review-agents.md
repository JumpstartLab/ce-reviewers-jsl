# AI/LLM-Based Code Review: What's Known About Making Reviewer Agents Effective

Research dimension for: how great technical code reviews are done, applied to a multi-agent review panel + orchestrator.

---

## 1. Key Findings

### 1.1 Noise is the #1 reason AI code review gets abandoned, and it's quantified across every tool

Every commercial and open-source LLM reviewer studied has a measurable false-positive/noise problem, and it is consistently cited as the top complaint:

- **CodeRabbit**: an independent 28-PR audit found 15% of comments were "useless/noise" and 21% were "nitpicking." CodeRabbit itself claims ~2 false positives per review run in third-party benchmarks — the best of the commercial tools tested. ([DevTools Academy state-of-AI-code-review-2025](https://www.devtoolsacademy.com/blog/state-of-ai-code-review-tools-2025), [CodeRabbit blog](https://www.coderabbit.ai/blog/how-coderabbit-delivers-accurate-ai-code-reviews-on-massive-codebases))
- **GitHub Copilot code review**: users report 15–25% of comments are incorrect, irrelevant, or too vague to act on; maintainers describe early rollouts as "too noisy," with most comments low-value until repo-specific custom instructions were added. ([GitHub Discussions #152385](https://github.com/orgs/community/discussions/152385), [Angie Jones](https://angiejones.tech/how-i-taught-github-copilot-code-review-to-think-like-a-maintainer/))
- **Cursor BugBot**: at GA (July 2025) only **52%** of flagged bugs were actually resolved before merge (implying ~48% were false positives or not worth fixing); by early 2026 that resolution rate had climbed to ~76–80% through iterative tuning — "15 points higher than the next-closest AI code review product." BugBot still over-flags nullable-type/null-check patterns and occasionally invents "swapped parameter" bugs. ([Cursor: Building a better Bugbot](https://cursor.com/blog/building-bugbot), [Cursor: Bugbot learning](https://cursor.com/blog/bugbot-learning))
- **Graphite Diamond**: claims sub-3% false-positive rate and >5% negative-feedback rate from developers on flagged comments, with an 82% "issue → actual fix" conversion in one reported sample. ([Graphite Diamond launch](https://graphite.com/blog/series-b-diamond-launch), [devclass](https://www.devclass.com/ai-ml/2025/03/19/graphite-debuts-diamond-ai-code-reviewer-insists-ai-will-never-replace-human-code-review/1626959))
- **Greptile**: highest bug-catch rate in a Greptile-run benchmark (82% vs. Bugbot 58%, Copilot 54%, CodeRabbit 44%, Graphite 6% — see caveat below) but also the noisiest: an independent benchmark found Greptile at 11 false positives per run vs. CodeRabbit's 2. Version 4 (March 2026) improved comment-acceptance rate from 30% to 43%. ([Greptile benchmarks](https://www.greptile.com/benchmarks), dev.to comparison below)
- **Qodo (formerly CodiumAI)**: shipped a Feb 2026 multi-agent architecture (separate bug/quality/security/test-coverage agents) that scored the highest published F1 (60.1%, +9pp over next-best) and highest recall (56.7%) in a cross-tool comparison. ([theaiagentindex](https://theaiagentindex.com/compare/codiumai-pr-agent-vs-qodo), [Qodo docs](https://docs.qodo.ai/code-review))

**Caveat on all vendor-run benchmarks**: Greptile's own benchmark ranks Greptile #1; CodeRabbit's own blog claims lowest FP rate; Qodo's marketing claims best F1. Treat single-vendor benchmark numbers as directional, not authoritative — methodology (which repos, what counts as "caught," severity weighting) varies and is chosen by the tool being favorably compared.

### 1.2 The most important independent data point: reviewers barely overlap, and AI comments are rarely acted on

Two independent, non-vendor studies matter more than any vendor benchmark:

- **A 3.5-week, 146-PR, 679-finding parallel run of CodeRabbit, Sentry Seer, Greptile, and Cursor BugBot** on a real production codebase (open-sourced data/tooling) found: **93.4% of findings were caught by exactly one reviewer — zero convergence across all four tools.** Each tool has a distinct "personality" (CodeRabbit: high-volume generalist, 3.4 findings/PR, 2.3% FP rate, but slow at 9.5 min; Sentry Seer: production-risk specialist, 6/6 correct on critical-severity, but 15% FP rate at high-severity; Greptile: zero false positives caught in this run but lowest finding density; BugBot: fastest and most predictable check-run behavior but tied to IDE-seat bundling). This is the strongest evidence in this research for **running multiple differently-focused reviewers is not redundant — it's necessary for coverage.** ([dev.to: Best AI Code Reviewer 2026 — 146 PRs](https://dev.to/_vjk/best-ai-code-reviewer-in-2026-we-ran-4-in-parallel-for-3-weeks-146-prs-679-findings-1c0f))
- **"Does AI Code Review Lead to Code Changes? A Case Study of GitHub Actions"** (arXiv 2508.18771) analyzed 22,326 AI review comments across 178 repos and 16 AI review tools. Human reviewers get **60%** of valid comments addressed; AI reviewers get **0.9%–19.2%** depending on tool (best: coderabbitai/ai-pr-reviewer at 19.2%; worst: mattzcarey/code-review-gpt at 0.9%). Success factors, with effect sizes: hunk-level granularity beat file-level; manual triggering (12.8%) beat automatic (6.8%); high code-to-text ratio in the comment (>0.5 ratio → 23.2% addressed) beat prose-heavy comments; comments aimed at inexperienced contributors were addressed at 16% vs. 3.3% for experienced devs (1,013+ prior commits) — and tellingly, **human reviewers concentrate 79% of their feedback on newcomers while AI reviews indiscriminately**, which is itself a signal-quality gap. ([arXiv 2508.18771](https://arxiv.org/html/2508.18771v1))

### 1.3 More review passes do not reliably mean better review

A 2026 paper, **"More Rounds, More Noise: Why Multi-Turn Review Fails to Improve Cross-Context Verification"** (arXiv 2603.16244), found that single-pass LLM verification outperformed multi-turn variants: additional review rounds increase recall only modestly while generating false positives at a much higher rate, destroying precision — attributed to context contamination (earlier-round outputs become false anchors) and inconsistency amplification across turns. This directly complicates the naive "just add another verification pass" instinct — the *design* of the second pass (independent, differently-scoped, adversarial) matters far more than simply adding rounds. ([arXiv 2603.16244](https://arxiv.org/pdf/2603.16244))

By contrast, a **stage-gated, single extra pass with an explicitly adversarial role** does work — see §2.3.

### 1.4 Diff-only context is measurably worse than augmented context

- A June 2026 study on Go code review (arXiv 2606.01859) tested three context-augmentation strategies against a diff-only baseline, measuring "RefineEM" (a refinement exact-match metric): neighboring-function context (+1.79pp), LSP-based semantic context via getDefinition/getReferences (+1.88pp), and IR-based "similar co-change" retrieval — code from files historically modified together (+2.16pp, the strongest single lever). The best *combination* (neighboring + similar-co-change) reached +3.76pp over no-context; notably, **adding semantic (LSP) context on top of that combination made things worse** (25.59% → 24.87%), evidence that indiscriminately piling on context can dilute signal. Their full pipeline (context augmentation + issue-list generation, i.e., "list everything, don't just report the top issue") reached 28.00% RefineEM vs. a 17.15% primary-issue-only baseline, approaching a 36.09% human-oracle ceiling. ([arXiv 2606.01859](https://arxiv.org/html/2606.01859v1))
- CodeRabbit's production architecture corroborates this qualitatively: before any LLM call, it runs 30+ static analyzers, builds an AST dependency graph ("Codegraph") from commit-history co-change patterns, queries a semantic vector index of past functions/PRs/tests ("Code Index" — "searches by purpose, not just keywords"), and injects team-specific style guidelines — only then does a reasoning model review, followed by a **verification agent that runs grep/ast-grep to ground each finding in actual code before posting**. ([CodeRabbit: accurate reviews at scale](https://www.coderabbit.ai/blog/how-coderabbit-delivers-accurate-ai-code-reviews-on-massive-codebases))
- Retrieval-augmented review generation is an active research area (LAURA, arXiv 2512.01356) combining context augmentation + retrieval of past review "exemplars" + systematic guidance, but a companion paper — **"When More Retrieval Hurts: Retrieval-Augmented Code Review Generation"** (arXiv 2511.05302) — is a direct warning label: naive RAG over-retrieval degrades review quality, echoing the LSP-context finding above. Curate context; don't maximize it.

### 1.5 Hallucination is real, and layered defenses reduce it more than any single fix

Estimates cited: 40–45% of raw AI-generated code contains security flaws, and ~19.7% of AI-recommended packages are fabricated/non-existent (general code-gen hallucination stats, not review-specific, but load-bearing for why *verification* steps matter in review agents). A cited Stanford study found combining RAG + RLHF + guardrails produced a 96% hallucination reduction vs. baseline. A useful five-layer mitigation stack: static analysis at the input layer → RAG with real code context (reported 60–80% reduction alone) → chain-of-thought generation → multi-agent cross-validation → deterministic output guardrails. A 2025 systematic literature review (arXiv 2511.00776) frames this as a maturing subfield with mitigation methods and known failure taxonomies. ([diffray.ai](https://diffray.ai/blog/llm-hallucinations-code-review/), [arXiv 2511.00776](https://arxiv.org/pdf/2511.00776))

### 1.6 Sycophancy: LLM reviewers are biased toward agreeing with the author

Multiple studies document that RLHF-trained models systematically upweight responses that align with a user's apparent beliefs — a direct threat to code review, where the "user" (PR author) has an obvious stake in the code being fine. Concrete mitigations found in practitioner writing: don't reveal whose code it is or that you wrote it ("review my coworker's PR" framing outperforms "review my PR"); ask for pros/cons rather than "is this the best way to do X," since the latter invites validation-seeking; and structurally, keep the reviewer prompt from ever seeing PR-author commentary/self-justification before it forms an initial judgment. ([Pareto Software: mitigating sycophantic bias](https://www.paretosoftware.fi/en/blog/mitigating-sycophantic-bias-in-llms), [arXiv 2603.18740: contextual bias in LLM-assisted security code review](https://arxiv.org/pdf/2603.18740))

### 1.7 LLM-as-judge (relevant to any synthesis/gating stage) has known, well-studied biases

Position bias (order of presentation swings verdicts), verbosity bias (judges reward longer responses as a quality proxy — worse in multimodal judges), and self-preference/family bias (a model — or a model from the same lab — rates its own/kin outputs higher). Mitigation with the most support: **ensemble judging across heterogeneous models**, plus randomizing/counterbalancing presentation order, plus explicit length-normalization (imperfect — it introduces its own distortion). This is directly relevant to any orchestrator/judge stage that scores or ranks findings from multiple reviewer agents. ([Self-Preference Bias in LLM-as-a-Judge, arXiv 2410.21819](https://arxiv.org/html/2410.21819v1), [Position Bias in LLM Judges](https://mbrenndoerfer.com/writing/position-bias-in-llm-judges))

## 2. Prompting / Architecture Patterns That Work (With Evidence)

### 2.1 Explicit "what NOT to flag" sections outperform positive-only instructions

Cloudflare's production system (7 specialized reviewer agents, 131,246 review runs over 30 days, open write-up) states plainly: *"Telling an LLM what not to do is where the actual prompt engineering value resides."* Every one of their sub-reviewers carries an explicit exclusion list. Result: **1.2 findings per review on average** across 159,103 total findings — deliberately low, biased hard for signal over noise. ([Cloudflare blog: Orchestrating AI Code Review at Scale](https://blog.cloudflare.com/ai-code-review/))

The G-Research write-up independently confirms the "DO NOT" pattern: without explicit negative instructions, "LLMs love to comment on formatting and naming, which are almost always noise." ([singhajit.com](https://singhajit.com/building-code-review-assistant-with-llms/), [G-Research](https://www.gresearch.com/news/building-a-code-review-tool-the-llm-patterns-that-actually-work/))

### 2.2 Specialized single-concern agents beat one general reviewer — both by architecture and by empirical coverage

- Cloudflare's production architecture: 7 distinct roles (security, code quality, performance, documentation, release management, compliance/"Engineering Codex," and an "AGENTS.md currency" reviewer), each with narrow scope and its own exclusion list, run in parallel and merged by a coordinator.
- Qodo's Feb 2026 rearchitecture from single-pass to multi-agent (bug / quality / security / test-coverage) produced the best published F1/recall in a cross-tool comparison.
- The 146-PR independent parallel study (§1.2) shows the mechanism *why* this works empirically: different tools/approaches literally see different bugs (93.4% single-tool-only hits) — a generalist single agent structurally cannot have the "attention budget" to catch everything a security-focused, performance-focused, and correctness-focused reviewer would each catch independently.
- Academic peer-review literature reaches the same conclusion by a different route: a PRISMA-aligned multi-agent system with one specialized agent per checklist item hit 84% agreement with expert-annotated scores; general "monolithic" LLM peer reviewers are described as "largely unvalidated" with chain-of-thought only buying 3.9–17.9% gains. ([emergentmind: Reviewer Agents](https://www.emergentmind.com/topics/reviewer-agents))

**Design implication**: granularity should track *distinct failure modes with distinct evidence types* (security needs taint/data-flow reasoning; performance needs complexity/query-pattern reasoning; correctness needs state/edge-case reasoning) — not arbitrary headcount. Cloudflare's 7 roles is a reasonable reference point for "how many personas before returns diminish," though their code-quality reviewer alone produced 47% of all findings, suggesting even within Cloudflare's split there's room to fragment "code quality" further if it's disproportionately the noisy bucket.

### 2.3 A second, differently-scoped, adversarial pass beats a bigger single-pass — but "just add rounds" doesn't

Two threads of evidence, read together, are the sharpest actionable finding for this project:

- **"Refute-or-Promote"** (arXiv 2604.19049): a stage-gated architecture with a **Proposer** (generates candidate defects), a **Refuter** that operates under deliberate **context asymmetry** (sees *less* than the proposer, so it can't simply rubber-stamp — it has to independently re-derive whether the defect holds), a **Promoter** that argues for validity only when refutation fails, a **Cross-Model Critic** (a *different* model family adjudicates deadlocks, countering single-model self-preference bias per §1.7), and explicit **kill mandates** — hard termination conditions once evidence conclusively resolves a claim, preventing infinite adversarial loops. This is architecturally the opposite of "run the same reviewer again": the second pass has a *different information diet* and a *different objective* (disprove, not confirm).
- **G-Research's two-pass pattern**: pass 1 optimizes recall (cast a wide net), pass 2 is a *separate LLM call* whose only job is to filter pass-1 findings for genuine rule violations against a machine-readable "rules index" that is the sole source of truth. This separation-of-concerns (generate broadly, then verify narrowly against ground truth) is explicitly credited with fixing the false-positive problem that a single complex prompt could not.
- **Contrast with §1.3**: "More Rounds, More Noise" found that *naive* multi-turn review (same agent, same objective, re-reading its own prior output) degrades precision as rounds increase — context contamination and self-reinforcing errors accumulate rather than cancel out. The distinguishing factor across these three sources: a verification pass only helps when it is **adversarial or independently-scoped** (different information, different objective, ideally different model), not merely iterative.

**Design implication**: don't add a "review the review" loop naively. Add a verifier with a narrower mandate (disprove/ground only), asymmetric information, and — where feasible — a different underlying model, with an explicit stop condition.

### 2.4 Structured output + a ground-truth index prevents drift

G-Research: JSON-schema-constrained output (Pydantic), validated against a canonical "rules index" mapping rule IDs → metadata, so "the model can suggest, but never define" what counts as a violation. This is the same idea as CodeRabbit's grep/ast-grep grounding step — the LLM's claim must resolve against something checkable, not just be internally coherent-sounding.

### 2.5 Confidence thresholds and asymmetric precision/recall targets by severity tier

G-Research's regression-test design used **asymmetric thresholds by rule severity**: 100% recall required for MUST/MUST-NOT (blocking) rules, 75% recall acceptable for SHOULD (advisory) rules, with an overall >85% precision bar and a hard zero-false-positive target for the highest tier. Separately, practitioner guidance recommends starting a numeric confidence threshold around 0.7 and tuning against observed false-positive rate, with low temperature (~0.2) for determinism — a "reliable-feeling" reviewer is explicitly linked to low run-to-run variance, not just accuracy. A cited (informal) technique — running the same diff through two different model families and only surfacing findings both flag — is reported to cut false positives ~60% at 2x inference cost, which is a cheap approximation of the Cross-Model Critic pattern in §2.3 for teams that don't want to build a full stage-gated pipeline.

### 2.6 Orchestrator/coordinator as synthesis-and-gate, not just aggregator

Cloudflare's coordinator (a stronger/more capable model — Opus-tier — reading the output of all 7 sub-reviewers) does three jobs, not one: (1) deduplicate and re-file findings into correct categories, (2) apply a "reasonableness filter" that actively re-reads source when uncertain and drops speculative findings — i.e., the orchestrator itself re-verifies, it doesn't just relay, (3) apply an explicit approval-rubric state machine (all-clear/trivial → approve; suggestions-only or non-production-risk warnings → approve-with-comments; multiple warnings → minor-issues/unapprove; anything critical or safety-risk → request-changes/block) that **biases toward approval** unless something concrete is wrong. This "approve unless proven risky" default, plus a human "break-glass" override used in only 0.6% of MRs, is presented as core to keeping the tool trusted rather than adversarial to shipping velocity.

### 2.7 Efficient context sharing beats re-deriving context per agent

Cloudflare writes per-file patches to disk once and has all sub-reviewers read from a shared `shared-mr-context.txt` cache rather than re-embedding the full diff into every agent's prompt — yielding an 85.7% prompt-cache hit rate across 120B tokens/month, "saving an estimated five figures" vs. naive per-agent full-context prompting. For a multi-persona panel, this is a concrete, load-bearing engineering pattern, not just a cost optimization: it also means every persona reviews the *same* grounded context rather than independently (and inconsistently) reconstructing it.

### 2.8 Risk-tiering controls both cost and (implicitly) noise

Cloudflare gates how many reviewer agents even run based on diff size/file-count risk tier: trivial diffs get 2 agents (~$0.20), "lite" diffs get 4 (~$0.67), and only diffs that are large or touch security-sensitive files get the full 7-agent treatment (~$1.68). This is a useful analog for a quality-review panel: not every change needs the full roster; matching persona-count to change risk avoids paying the noise cost of an over-staffed review on a one-line fix.

### 2.9 Anthropic's own guidance: prefer the simplest structure, escalate deliberately

From "Building Effective AI Agents": start with simple prompts, add multi-step/multi-agent structure "only when simpler solutions fall short." Two named patterns map directly onto a review panel:
- **Evaluator-optimizer**: one LLM generates, another evaluates/critiques in a loop — fits when "responses are demonstrably improved when a human [or evaluator] articulates feedback" and the evaluator LLM can actually produce that feedback (their example: literary translation, where nuance-checking is a distinct skill from generating). This maps onto having a critique/gating stage be architecturally separate from the generation (finding) stage.
- **Orchestrator-workers**: fits when subtasks *can't be predicted in advance* — the orchestrator decides what specialist work is needed per input, rather than always running a fixed pipeline. This argues for an orchestrator that can, e.g., skip the performance persona on a docs-only change (echoing Cloudflare's risk tiering) rather than always fanning out to every persona.
- On parallel voting for gating: "running the same task multiple times... multiple prompts evaluating different aspects or requiring different vote thresholds to balance false positives and negatives" is Anthropic's own framing of a confidence-gate built from redundancy.
- On tool/interface design generally: invest in the agent-computer interface as seriously as human UX — clear boundaries between tools, example usage, edge cases documented in the tool definition itself, and iterative testing of *how the model actually uses the tool* against a battery of example inputs. Applies directly to whatever interface reviewer personas use to read code/context and to emit findings.

([Anthropic: Building Effective AI Agents](https://www.anthropic.com/research/building-effective-agents))

## 3. Failure Modes and Mitigations

| Failure mode | Evidence | Mitigation(s) with evidence |
|---|---|---|
| **Noise / low signal-to-noise** drives abandonment | 15–25% of comments low-value across Copilot/CodeRabbit audits; "#1 reason teams abandon AI reviewers" per multiple practitioner sources | Explicit "what NOT to flag" sections (Cloudflare: 1.2 findings/review avg); cap findings per PR (practitioner guidance: ~8 forces prioritization); confidence thresholds (~0.7 starting point) |
| **Trivial/nitpick flooding drowns real findings** | "If an AI reviewer posts 20 comments with 18 trivial, developers ignore all 20, including the 2 real bugs" — ratio needs to stay above ~50% actionable to preserve trust | Severity taxonomy with a hard blocking/nitpick split (critical/warning/suggestion, or RFC-2119 MUST/SHOULD mapped to severity automatically rather than left to model judgment); route nitpicks to a separate low-priority channel, not inline with blockers |
| **Diff-only context misses real issues / produces superficial feedback** | Go-code study: context augmentation improved RefineEM by up to +3.76pp combined; CodeRabbit's production stack layers static analysis + AST graph + semantic index before any LLM call | Curated context (neighboring function, co-change history, semantic index of past PRs/tests, team style guide) — but stop adding context once returns diminish (over-adding LSP context on top of a good combination *hurt* in the Go study; "When More Retrieval Hurts" paper is a direct warning) |
| **Hallucinated findings** (fabricated bug, wrong line, invented API) | 40–45% of raw AI-gen code has security flaws; ~20% of package recommendations fabricated (general code-gen stats); HalluJudge paper targets exactly this for review | Ground every claim against executable evidence — CodeRabbit and G-Research both run grep/ast-grep or a rules-index lookup before a finding is allowed to post; layered defense (static analysis → RAG → CoT → cross-validation → guardrails) reported to compound toward ~96% reduction |
| **Sycophancy toward the PR author** | RLHF preference models upweight author-agreeable responses; documented in LLM-assisted security review specifically | Don't frame the code as "my" code in the reviewer prompt; ask "what are the pros/cons" rather than "is this good"; keep author self-justification/commit messages out of the initial-pass context so the first read is unbiased |
| **More review rounds ≠ better review** | Naive multi-turn re-review increases FPs faster than it increases recall (context contamination, self-reinforcing errors) | Make the second pass *adversarial and asymmetric* (different info, different objective, ideally different model) rather than "review your own review again"; add explicit kill mandates/stop conditions |
| **LLM-as-judge bias** in any gating/synthesis step | Position, verbosity, and self-preference bias well-documented, worse in multimodal judges | Ensemble judging across heterogeneous models; randomize/counterbalance presentation order fed to the judge; be suspicious of "longer finding = more credible" |
| **AI reviews indiscriminately vs. human reviewers who concentrate feedback where it's needed** | GitHub Actions study: humans put 79% of feedback on newcomers; AI tools show no such targeting, and AI comment-addressed rate (0.9–19.2%) is far below human (60%) | Feed the reviewer author-context (is this a new contributor? is this a hot/fragile file?) so severity/tone can be calibrated, not just code content |
| **Single monolithic reviewer misses issues a specialist would catch** | 93.4% of findings in the 146-PR study were single-tool-only; zero convergence across 4 tools | Multiple narrowly-scoped personas, each with a distinct evidence type/failure mode, run in parallel rather than one generalist prompt trying to cover everything |
| **Cost/latency blowup from full-roster review on every change** | Cloudflare: full 7-agent review costs ~8x a trivial 2-agent review; 500-file diffs are expensive at 7 concurrent frontier-model calls | Risk-tier the roster by diff size/sensitivity — not every change needs every persona |
| **Findings not actually actioned even when correct** | GitHub Actions study: hunk-level + high code/text ratio + concise + manually-triggered reviews were addressed far more; file-level/verbose/auto-triggered were mostly ignored | Post at hunk/line granularity, favor code-diff-style suggestions over prose, keep comments concise, consider gating review triggering to be more deliberate rather than always-on |
| **Prompt injection via user-controlled content in the diff/PR description** | Cloudflare: had to strip XML boundary tags from user content after engineers tested exactly this against the internal tool | Sanitize/strip structural delimiters from any untrusted text (commit messages, PR descriptions, code comments) before it enters agent context |

## 4. Direct Recommendations for a Multi-Persona Quality-Review Panel + Orchestrator

**Roster granularity.** Use Cloudflare's ~5–8 persona range as a reference ceiling, but split by *distinct evidence type*, not by vague topic. A persona should own a kind of reasoning a generalist can't reliably do in the same pass (e.g., a security persona doing taint/data-flow reasoning is a different cognitive task than a correctness persona doing edge-case/state reasoning). If one persona (analogous to Cloudflare's "code quality," 47% of findings) turns out to dominate volume, split it further rather than letting it become the noise source. Risk-tier the roster: cheap/small changes get 2–3 personas, full changes get the whole roster; let the orchestrator decide which personas are even relevant per diff (Anthropic's orchestrator-workers pattern), rather than always fanning out to everyone.

**Gating / confidence.** Every persona prompt needs an explicit, itemized "do NOT flag" section — this is repeatedly identified as the single highest-leverage prompt-engineering lever for noise reduction. Adopt a severity taxonomy with an automatic (not model-judged) mapping where possible — e.g., MUST/MUST-NOT → blocking, SHOULD → warning, suggestion → nitpick — and set asymmetric precision/recall targets per tier (near-100% recall tolerated for blocking-tier candidates pre-verification, but a near-zero false-positive bar for what actually gets *posted* as blocking). Cap total findings surfaced per review (practitioner consensus lands around 5–10) to force prioritization.

**Verification / adversarial pass.** Don't just re-run the same persona a second time — that measurably increases noise (§1.3, §2.3). Instead, give each finding (or each candidate blocking finding, if you want to bound cost) a second, narrower pass whose only job is to *disprove* it: less context than the original finder (context asymmetry), a different framing ("try to show this is NOT a real issue"), and ideally a different underlying model to counteract self-preference bias. Ground the disproof attempt in executable checks (grep/ast-grep/actually running a linter) wherever the codebase allows it, not just re-reasoning in prose. Give this pass an explicit stop condition so it terminates on conclusive evidence rather than looping.

**Synthesis / orchestrator.** The orchestrator's job is dedup + re-categorize + apply a reasonableness filter (re-read source when uncertain, drop speculative claims) + apply an explicit decision rubric that is biased toward *not* blocking unless something concrete and severe is found — Cloudflare's approve/approve-with-comments/request-changes state machine is a good template, and their 0.6% break-glass override rate suggests this bias-toward-approval is what keeps a panel trusted rather than seen as an obstacle. If the orchestrator itself scores/ranks findings from multiple personas, be aware of and mitigate LLM-as-judge biases (position, verbosity, self-preference) — randomize presentation order into the orchestrator's context, and don't let finding length be a de facto credibility signal.

**Context feeding.** Curate, don't maximize. The evidence favors: the specific diff/hunk with enough neighboring-function context to resolve references; a lightweight dependency/co-change map (which files historically change together — the single strongest lever in the Go study); a searchable index of past PRs/review comments/rejected-findings so personas don't re-raise things already litigated; and an explicit style-guide/standards document treated as ground truth that findings must cite, not just gesture at. Stop there — over-adding semantic/LSP-style context on top of a good baseline measurably hurt results in at least one study, and a dedicated paper ("When More Retrieval Hurts") is a direct caution against maximal RAG. Keep author self-justification/commit-message rationale out of first-pass context to blunt sycophancy; reintroduce it, if at all, only at the orchestrator/synthesis stage.

**Output format.** Favor hunk/line-level inline comments with concrete diff-style suggestions over prose-heavy or file/PR-level summaries — this correlates strongly with whether developers actually act on a finding (GitHub Actions study: hunk-level and high code-to-text ratio were the two strongest predictors of a comment being addressed). Group by severity tier in the final synthesized review rather than presenting one flat list, and keep total surfaced findings low. Consider whether manual/deliberate review triggering (vs. always-on-every-push) fits the workflow — the same study found manually-triggered reviews were addressed at roughly double the rate of automatic ones, plausibly because manual triggering correlates with the author actually being ready to receive feedback.

**Engineering hygiene, not just prompting.** Share context across personas via a cache rather than re-deriving/re-embedding it per agent (Cloudflare's 85.7% cache-hit rate is a real cost lever, and also a consistency lever — all personas see the same grounded facts). Sanitize any untrusted text (PR descriptions, commit messages, in-code comments) for prompt-injection risk before it enters any persona's context. Use structured, schema-validated output for every finding (rule/category, severity, file:line, evidence, suggested fix) so the orchestrator can dedup and gate mechanically rather than by re-reading prose.
