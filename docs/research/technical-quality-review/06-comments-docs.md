# Comment Style/Content and Documentation Quality in Code Review

Dimension: how expert reviewers evaluate comments and documentation, and whether docs match actual functionality.

## 1. Key findings

### 1.1 The comments debate, steel-manned both ways

The debate has a clean primary source: John Ousterhout (*A Philosophy of Software Design*, "APOSD") and Robert "Uncle Bob" Martin (*Clean Code*) directly compared notes on where they disagree, published as a joint document. This is the single best artifact for understanding the real shape of the disagreement, not the strawman version either side usually gets reduced to.

- **[aposd-vs-clean-code](https://github.com/johnousterhout/aposd-vs-clean-code)** (Ousterhout & Martin's own comparison): Ousterhout's core claim is structural, not stylistic — *"without comments there is no way to have abstraction or modularity."* His argument: an interface comment is what lets a caller use a method without reading its implementation; delete the comment and you've deleted the abstraction boundary, because the caller now has to read the code to know what's safe to assume. He estimates missing/absent comments cost 10-100x more in lost productivity than occasionally-wrong comments do, and argues for roughly 5-10x more commenting than Martin's practice, with the default posture that readers should be able to trust a comment without re-deriving it from the code.
  Martin's position, in his own words from the exchange: *"Comments are always failures. We must have them because we cannot always figure out how to express ourselves without them."* and *"I don't trust comments to be maintained, nor do I trust that they will be read."* His remedy is to push information into code — better names, extracted methods with intention-revealing names, small functions — rather than into prose that can drift.
  Where they actually agree: comments are sometimes necessary, well-placed comments help, some information genuinely can't be expressed in code, and comments that restate the obvious are waste. The real disagreement is a probability/cost estimate — how often is a comment worth the maintenance risk — not a binary "comments good/bad."

- **[Ousterhout's chapter summaries](https://carstenbehrens.com/a-philosophy-of-software-design-summary/), [dev.to summary](https://dev.to/markadel/a-philosophy-of-software-design-summary-pk9), [Stanford extract PDF](https://web.stanford.edu/~ouster/cgi-bin/aposd2ndEdExtract.pdf)**: Ousterhout calls "if code is well-written, it doesn't need comments" a **"delicious myth"** — delicious because it excuses the reader from the work of commenting. His taxonomy:
  - **Interface comments** — go with the declaration (class, function, method): describe the *what*, behavior, arguments, return values, side effects, exceptions, and things the caller must satisfy — including non-obvious details like units, whether null is allowed, whether bounds are inclusive. These describe things the caller needs and cannot see by reading the signature alone.
  - **Implementation comments** — describe *how* the code works internally, at a level of abstraction usually a notch above the code itself (not restating line-by-line).
  - A named failure mode: **interface-comment contamination** — an interface comment that leaks implementation detail the caller doesn't need. This blurs the abstraction boundary the interface comment exists to create.
  - He advocates writing comments *before* the code (comment-first, similar spirit to README-driven development below) as a design tool, not an afterthought.

- **[Hillel Wayne, "The Myth of Self-Documenting Code"](https://buttondown.com/hillelwayne/archive/the-myth-of-self-documenting-code/)**: A sharper, more empirical steel-man against "just write clear code." Wayne distinguishes **tactical knowledge** (what does this code do, step by step — the thing clear naming *can* convey) from **operational knowledge** (how does this fit into the larger system, why does it exist, what does it *not* do — the thing naming structurally cannot convey). His example: in Uncle Bob's own FitNesse codebase, a clearly-named call chain requires "digging four methods deep" to find the actual API call being made — each individual method reads clean, but the operational fact of what actually happens is invisible without a comment naming it. He also flags that tests-as-documentation smuggle in a second responsibility (verification) that competes with clarity, and that "comment why not what" as a rule silently excludes whole categories of useful information — negative results ("we tried X, it doesn't work because Y"), performance/optimization rationale, warnings. His own verdict: comments "*are* garbo" but remain "the only means we have of interleaving code with expressive human-to-human communication" — the fix is better documentation tooling, not abandoning comments.

- **[Clean Code summary/critique sources](https://gist.github.com/wojteklu/73c6914cc446146b8b533c0988cf8d29), [In Defense of Clean Code](https://dev.to/thawkin3/in-defense-of-clean-code-100-pieces-of-timeless-advice-from-uncle-bob-5flk)**: Martin's chapter is more nuanced than "no comments" — he has an explicit taxonomy of *good* comments (legal, informative, explanation of intent, clarification, warning of consequences, TODO, amplification) versus *bad* ones (mumbling, redundant, misleading, mandated, journal comments, noise, position markers, closing brace comments, attributions, commented-out code, HTML in comments, too much information, inobvious connection, function headers as comment-substitute, non-public API docs). The load-bearing claim that critics push back on is narrower than "never comment": *"Inaccurate comments are far worse than no comments at all"* and comments should be a last resort after trying to express the same information in code (rename, extract function, extract variable).

**Steel-manned synthesis**: this is not "comments good" vs. "comments bad." It's a dispute over (a) how much information genuinely cannot be pushed into code/names/tests (Ousterhout and Wayne say: more than Clean-Code culture assumes — abstraction boundaries and operational/system context specifically resist encoding into names), and (b) how much to weight decay risk against the value of what a good comment captures (Martin weights decay risk higher and pushes for code-native expression wherever physically possible). A reviewer persona should hold both: demand comments where code structurally cannot carry the information (interfaces, why, invariants, units, non-obvious constraints, negative results) and push back on comments that merely restate what a rename would already say.

### 1.2 Comment taxonomy

- **Why vs. what** ([Stack Overflow Blog best practices](https://stackoverflow.blog/2021/12/23/best-practices-for-writing-code-comments/), [MIT Comm Lab](https://mitcommlab.mit.edu/broad/commkit/coding-and-comment-style/)): the most repeated heuristic across sources — comments should carry information the code cannot: rationale, intent, business context, why an alternative was rejected, non-idiomatic choices. "What" comments that restate the next line are near-universally flagged as noise. Corollary heuristic worth keeping distinct from "why vs what": *if you can't write a clear comment, that's often a sign the code itself is unclear* — comment difficulty is a code-smell detector, not just a prose problem.
- **Interface vs. implementation comments** (Ousterhout, above) — a structurally different axis than why/what. Interface comments serve *callers* (abstraction boundary); implementation comments serve future *maintainers* of this exact function.
- **Martin's granular taxonomy of good/bad comment types** (Clean Code, above) is the most complete practical checklist a reviewer can use.
- **TODO/FIXME hygiene** ([Aikido](https://www.aikido.dev/code-quality/rules/how-to-remove-lingering-todo-and-fixme-comments-from-your-codebase), [ACM TOSEM paper "What Makes a Good TODO Comment?"](https://dl.acm.org/doi/10.1145/3664811)): concrete empirical numbers — **46.7% of TODO comments in open-source repos are low-quality** (ambiguous, uninformative, or dead), and the **average time from TODO introduction to removal is 166.31 days**. Practical hygiene rule that recurs across sources: a TODO without an owner, a ticket reference, or a rationale is close to worthless — good practice is a linter/CI rule that rejects TODOs lacking a ticket number, and code review should treat unresolved TODOs on the current diff as a blocking item to either finish, ticket, or justify.
- **Commented-out code** ([Nayuki](https://www.nayuki.io/page/dont-share-commented-out-code), [Kent C. Dodds](https://kentcdodds.com/blog/please-dont-commit-commented-out-code), [HN discussion](https://news.ycombinator.com/item?id=7518304)): near-unanimous — delete it, don't comment it out, because (a) it produces poor diffs (comment markers show as changes on every line even though nothing executes differently), (b) it's unreadable noise future readers must consciously skip, (c) only the author who commented it out knows why/how-long/what-replaced-it, and (d) version control already remembers deleted code, so "keeping it just in case" is a solved problem via `git log`/branches, not comments.

### 1.3 Empirical work on comment decay / code-comment inconsistency

Verifiable, cited findings (all with methodology described, not blog claims):

- **[arXiv:2409.10781 — "Investigating the Impact of Code Comment Inconsistency on Bug Introducing"](https://arxiv.org/html/2409.10781v1)**: Studied 32 Apache Java repositories using GitCProc (bug-fixing commit identification) + the SZZ algorithm (bug-introducing commit identification), with odds-ratio statistical testing. Finding: commits with code-comment inconsistencies were **~1.5x more likely** to be bug-introducing within a 7-day window (dropping to ~1.14x at 14 days — the risk is highest right after the inconsistency is introduced and decays). Same paper also benchmarks a fine-tuned GPT-3.5 model for detecting inconsistencies (94.4% precision / 88.3% recall against a verified test set), relevant for automated doc-drift detection tooling.
- **[arXiv:2307.04291 — "Wait, wasn't that code here before? Detecting Outdated Software Documentation"](https://arxiv.org/abs/2307.04291)**: Built an automated tool (as a GitHub Action) that scans documentation for references to code elements (functions, files, symbols) that have since moved/renamed/been deleted. Headline finding: **more than a quarter of the 1,000 most popular GitHub projects contained at least one outdated code reference in their documentation** — i.e., doc-code drift is not a fringe problem even in well-resourced, popular OSS projects.
- **[arXiv:2403.00251 — "Are your comments outdated? A method for code-comment consistency detection"](https://arxiv.org/html/2403.00251v1)** and **[arXiv:2506.20558 — CCISolver](https://arxiv.org/pdf/2506.20558)**: further ML-based detection work confirming this is an active research area with dataset-backed benchmarks (e.g., the CUP2 dataset used across multiple papers), not just anecdote.
- General code-comment co-evolution study cited across searches: **97% of comment changes happen in the same commit/revision as the corresponding code change** when they happen at all — but newly-added code is frequently under-commented in the first place, meaning the drift problem is as much about comments never being written as about them going stale after the fact.

Caveat: treat the specific precision/recall/odds-ratio numbers as indicative of "this is a real, measurable, non-trivial problem," not as universal constants — they're single-study results on specific corpora (Apache Java repos, GitHub's top 1000).

### 1.4 Documentation frameworks

- **[Diátaxis](https://diataxis.fr/)** ([start-here](https://diataxis.fr/start-here/)): the most cited modern documentation framework, adopted by projects like Gatsby, Django, Cloudflare. Splits documentation into four kinds by the reader's need at the moment they reach it, on two axes (action vs. cognition, and study vs. work):
  - **Tutorial** — learning-oriented, practical, instructor takes responsibility for the learner's success ("teach me to make a thing")
  - **How-to guide** — goal-oriented, practical, assumes competence, addresses a real task ("help me do this specific thing")
  - **Reference** — information-oriented, theoretical, accurate/complete/free of narrative ("tell me the facts")
  - **Explanation** — understanding-oriented, theoretical, discursive ("help me understand why/how this works")
  A reviewer can use Diátaxis as a *classification check*: is this doc change mixing modes (e.g., a reference page that suddenly has tutorial narrative, or a how-to that doesn't actually solve a task)? Mixing modes is the framework's named anti-pattern — it degrades all four use cases at once.
- **[Google Technical Writing courses](https://developers.google.com/tech-writing)** (Technical Writing One/Two, public since 2020): core teachable heuristics — state the document's scope and audience up front; short sentences are more powerful than dense ones; adopt a single house style guide rather than inventing one; provide navigation/signposting (TOC, "what's next" links) so readers can self-locate. Useful as a rubric for reviewing prose-heavy docs (READMEs, guides), distinct from API reference review.
- **[README-driven development](https://tom.preston-werner.com/2010/08/23/readme-driven-development)** (Tom Preston-Werner, GitHub co-founder): write the README *before* the code — "until you've written about your software, you have no idea what you'll be coding." Functionally the same design-tool argument Ousterhout makes for comment-first design, applied at the project/library level instead of the function level. Byproduct: the README never goes stale relative to intent because it *is* the spec, written when motivation and clarity are highest, not backfilled after the fact under time pressure.

### 1.5 Doc-code drift: keeping docs matched to functionality

- **Docs-as-code + same-PR review** ([DeepDocs](https://deepdocs.dev/documentation-review-process/), [DraftView](https://www.draftview.app/blog/docs-as-code-review-workflow-complete-guide)): the dominant modern practice — documentation lives in the same repo, same version control, same PR, same review as the code change it describes. The specific trigger condition worth encoding for a reviewer: *if a change alters how someone integrates, configures, or debugs the system, the docs are part of the same changeset, whether or not they live in the same folder.* This reframes "should docs be updated" from a vague nice-to-have into a mechanical check: does this diff change public behavior, config surface, CLI flags, or error conditions? If yes and docs weren't touched, that's a reviewable gap.
- **Doctests / executable documentation** ([Python doctest docs](https://docs.python.org/3/library/doctest.html), [Real Python on doctest](https://realpython.com/python-doctest/), Rust doctests, `markdown-doctest`): the strongest mechanical anti-drift technique — examples embedded in docstrings/docs are executed as tests; if the example's output no longer matches reality, CI fails. Policy pattern worth citing: *"every public utility function should have at least one executable example; CI runs doctests; if docs examples fail, docs don't ship."* This converts "doc accuracy" from a judgment call into a build gate for the subset of documentation that's example-shaped (which is most API reference documentation).
- **Doc coverage tooling**: `docstr-coverage` (Python), Sphinx's `coverage` extension, Compodoc's `--coverageTest` / `--coverageTestThresholdFail` (Angular), and a JS `doc-coverage` tool — all measure *presence* of documentation (function/class/module has a docstring at all) and can gate CI on a coverage threshold. Important limitation for a reviewer persona: these tools catch *absence*, not *inaccuracy* — a docstring that exists but lies passes doc-coverage checks. Presence and correctness are different problems requiring different tools (coverage tools for the former, doctest/consistency-detection ML models for the latter).
- **API breaking-change detection**: tools like `oasdiff` diff OpenAPI spec versions and flag breaking changes automatically; the broader principle repeated across sources is that *observable behavior* (timing, error text, ordering) can become a de facto contract even when not part of the formal spec — so "did this change something documented" needs to include implicit/observed contracts, not just the explicit API surface.

### 1.6 API reference quality — Stripe as exemplar

- **[Stripe API docs](https://docs.stripe.com/api)**, analyzed in [WriteChoice's teardown](https://writechoice.io/blog/best-api-documentation-stripe-teardown) and [apidog's analysis](https://apidog.com/blog/stripe-docs/): concrete, citable patterns that make reference docs "great" rather than merely "complete":
  - **Examples-first**: quickstart gets a developer to a working call in under 10 minutes; every reference entry has runnable code samples in multiple languages, switchable with one click, that actually execute against the current API version (not hypothetical pseudocode).
  - **Inline definition on first use**: unfamiliar concepts (e.g., "idempotency key," "PaymentIntent") get a one-sentence definition the first time they appear, inline, so the reader never has to context-switch to a glossary page.
  - **Structured, actionable error documentation**: errors carry a machine-readable `code`, a human-readable `message`, and a `doc_url` pointing straight at the relevant explanation — plus errors are grouped by *category* (card errors, API errors, auth errors, rate limits) with guidance on how the *developer* should handle each category programmatically and what the *end user* should see, not just an HTTP status list.
  - **Action-oriented headings**: "Create a customer," "Accept a payment" — headings describe the task accomplished, not the noun being discussed.
  This is a usable rubric: does a new/changed API doc entry have a runnable example, inline definitions of new terms, and does the error documentation say what the caller should *do*, not just what went wrong?

### 1.7 Changelogs and commit messages as documentation

- **[Conventional Commits](https://www.conventionalcommits.org/en/v1.0.0/)**: structured `type(scope): description` format (feat/fix/perf/refactor/chore/docs/etc.) that makes commit history machine-parseable, enabling automated changelog generation and semantic version bumps. The discipline side-effect worth noting for reviewers: *if a commit can't be cleanly typed, it's often doing too many unrelated things at once* — this makes the commit-message convention double as a scope-creep detector during review, not just a formatting rule.
- Good commit message practice generally: specific over vague ("Fix null pointer in user validation" vs. "fix bug") — this is the same why-not-what principle from comments, applied to commit messages as a documentation artifact in their own right (git blame / git log is documentation).

## 2. Concrete heuristics for reviewing comments and docs

1. **Ask "could a rename fix this?"** before accepting a what-comment. If the comment just restates the next line, the fix is a better identifier, not the comment (Clean Code's core mechanism).
2. **Ask "could code fix this?" only up to the abstraction boundary.** Beyond it (interface behavior, invariants, units, side effects, why an alternative was rejected, operational/system-level context), no amount of renaming substitutes — flag missing comments there as a real gap, not a style nit (Ousterhout, Wayne).
3. **Check interface comments for contamination.** If a public interface's comment describes internal implementation details the caller doesn't need, that's a modularity leak worth flagging even though "more docs" is usually good.
4. **Treat units, nullability, and boundary-inclusivity as mandatory interface-comment content** for any non-trivial parameter or return value — these are exactly the class of thing code cannot self-document and are cheap to get wrong silently.
5. **Flag TODO/FIXME without an owner, ticket, or rationale** as blocking, not stylistic — cite the empirical 46.7%-low-quality / 166-day-lifetime data as the reason this matters rather than treating it as taste.
6. **Flag commented-out code for deletion**, full stop — version control is the only legitimate home for "code we might want later."
7. **Cross-check the diff against docs mechanically**: does this PR change a public function signature, CLI flag, config key, error condition, or default value that is described in README/reference docs, and were those docs touched in the same diff? If not, that's a specific, nameable gap — not a vague "docs might need updating" comment.
8. **For example-bearing docs (docstrings, README snippets, tutorials), ask whether the example is or could be executed in CI** (doctest-style). If it's illustrative code that's never run, treat it as an unverified claim more than as documentation.
9. **For error/exception documentation, check it's actionable**: does it say what the caller should do, not just what went wrong (Stripe standard)? Category grouping and a doc link per error type beats a flat status-code list.
10. **When reviewing prose docs, apply Diátaxis mode-check**: is this a reference page suddenly narrating a tutorial, or a how-to that doesn't resolve to a concrete task? Mixed-mode docs read worse for every audience even when individually well-written.
11. **Use the same why-not-what and specificity bar for commit messages** as for comments — a commit message is documentation with a permanent URL (`git blame`).
12. **Weight comment-decay risk by proximity, not just existence.** The bug-risk research shows inconsistency risk peaks shortly after introduction — comments on recently-changed, actively-touched code deserve the most scrutiny in review, more than comments on stable, rarely-touched code.

## 3. Anti-patterns

- **"Good code doesn't need comments"** used as a blanket excuse to skip interface documentation — Ousterhout's "delicious myth"; conflates "this function's internals are readable" with "the caller knows the contract."
- **What-comments / restating the code** — `i++; // increment i`.
- **Comment as apology for bad naming/structure** — the comment is doing the job a rename or extraction should do (Clean Code's central complaint, legitimate as far as it goes).
- **Stale/misleading comments left uncorrected** — worse than no comment, because it actively misdirects and the reader has spent trust on it (near-universal agreement across every source, including both sides of the Ousterhout/Martin debate).
- **Commented-out code** — noise, breaks diffs, orphaned context nobody but the original author understands.
- **Vague TODOs with no owner/ticket/date** — functionally permanent, decays into landscape noise (166-day average lifetime, ~47% low quality).
- **Docs updated in a separate, later PR (or never)** — breaks the same-changeset principle; creates a window where docs are actively wrong, not just incomplete.
- **Doc coverage mistaken for doc correctness** — a tool reporting 100% docstring coverage says nothing about whether those docstrings are true.
- **Interface-comment contamination** — leaking "how" into a comment whose job is "what," which defeats the abstraction the comment exists to provide.
- **Illustrative-only code examples in docs that are never executed/tested** — silently rot the moment the API changes underneath them; the more polished the example looks, the more a reader trusts it, which makes the failure worse.
- **Mode-mixing in structured docs** (Diátaxis) — tutorial narrative inside reference pages, or reference-style exhaustiveness inside a how-to, degrading both.

## 4. Implications for an AI comments/docs reviewer persona

**What can be detected mechanically (high confidence, cheap):**
- Commented-out code blocks (pattern match on comment syntax wrapping otherwise-valid code).
- TODO/FIXME without a ticket reference or owner tag (regex + lookup against issue tracker).
- Public function/class/method signatures changed in the diff with no corresponding change to their docstring/interface comment in the same diff (AST diff + comment-block diff, paired by symbol).
- Public API surface (exported functions, CLI flags, config keys, HTTP routes) changed with no corresponding change to README/reference docs in the same PR (requires a symbol → doc-location index, buildable from doctest/coverage tooling patterns above).
- Presence/absence of docstrings against a coverage baseline (docstr-coverage-style), to flag *newly undocumented* public symbols specifically — presence of pre-existing gaps is lower priority than a *newly introduced* public symbol shipping bare.
- Example code blocks in docs/docstrings that fail when executed (if a doctest/example-runner is wired into CI) — this is a binary pass/fail signal, not a judgment call.
- Stale symbol references in prose docs (function/file names mentioned in markdown that no longer exist in the codebase) — same technique as the arXiv:2307.04291 tool.

**What remains a judgment call (requires reading intent, not just diffing):**
- Whether a given comment is a *why* comment or a disguised *what* comment — this needs semantic understanding of whether the comment's content is derivable from the code next to it. A cheap heuristic proxy: does the comment contain a rationale word/clause ("because," "since," "to avoid," "workaround for," a ticket link) vs. purely describing control flow in English. Not airtight, but a reasonable first-pass signal before a closer read.
- Whether an interface comment is *contaminated* with implementation detail the caller doesn't need — requires knowing who the audience of the comment is (caller vs. maintainer), which isn't syntactically marked in most languages.
- Whether the *content* of an updated doc/comment is actually still true, when no executable example exists to check it against — this is the hard case. Best available lever: flag it for closer read when the code logic the prose describes has measurably changed (e.g., a changed conditional, changed return type, changed error path) near a comment/doc block that wasn't touched — a *heuristic* proxy for likely staleness, not proof.
- Whether a TODO is "vague" vs. "sufficiently scoped" — length and presence of a ticket link are weak proxies; genuinely judging actionability needs the same read a human reviewer would give it.
- Diátaxis mode-mixing — requires understanding the *purpose* of the page a doc change lives on, not just its content, since the same sentence can be fine in one doc type and wrong in another.
- Whether missing documentation is actually costly (Ousterhout's "abstraction boundary" test) vs. merely missing but harmless — depends on whether the thing being documented is actually consumed by other code/people beyond its author, which requires call-site / usage awareness, not just presence-of-comment awareness.

**Practical design for the persona**: run the mechanical checks first as cheap, high-confidence, always-on gates (commented-out code, orphaned TODOs, signature-changed-without-comment-changed, doc-referenced-symbol-doesn't-exist, example-doesn't-execute). Reserve model judgment for the narrower, more expensive question of *content* quality — is this specific comment/doc block saying something true, non-obvious, and worth the maintenance cost — and scope that judgment pass to the lines the mechanical pass already flagged as touched-and-risky, rather than re-reading every comment in a diff at the same depth. This mirrors the empirical finding that decay risk is highest immediately after a change: bias review attention toward comments/docs adjacent to logic that changed in *this* diff, not toward auditing unrelated stable code.
